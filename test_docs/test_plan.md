## 1. Введение

Документ описывает стратегию и объём ручного тестирования REST API интернет-магазина. 
Сервис реализован на FastAPI и работает поверх PostgreSQL с пулом соединений. 
Цель тестирования — убедиться, что API корректно обрабатывает валидные и невалидные данные, возвращает правильные коды ответа и не нарушает целостность данных.

## 2. Объект тестирования

Тестируемая система (SUT) — REST API магазина с сущностями **товары (products)** и **заказы (orders)**. 
Отдельного ресурса пользователей нет — `user_id` фигурирует только как поле при создании заказа.



| Метод | Endpoint | Назначение |
|---|---|---|
| GET | `/health` | Проверка доступности сервиса |
| GET | `/products` | Список товаров (схема ответа `ProductOut`) |
| GET | `/products/{product_id}` | Карточка товара по ID |
| POST | `/orders` | Создание заказа (тело `OrderIn`, ответ `OrderOut`) |

Тело создания заказа (`OrderIn`) — плоское, один товар на заказ:

```json
{ "user_id": 1, "product_id": 3, "quantity": 1 }
```

## 3. Объём тестирования (Scope)

**В объёме (In scope):**
- Функциональные проверки чтения товаров и создания заказов.
- Валидация входных данных заказа (обязательные поля, типы, граничные значения `quantity`).
- Негативные сценарии (несуществующие ID, некорректное тело запроса, неверные методы).
- Проверка кодов состояния HTTP и структуры ответа.
- Бизнес-логика заказа (подсчёт итоговой суммы, проверка остатков).

**Вне объёма (Out of scope):**
- Нагрузочное и стресс-тестирование — покрыто отдельным проектом (Locust).
- Пентест и глубокий security-аудит.
- Тестирование инфраструктуры (Docker, БД) как отдельных компонентов.

## 4. Подход к тестированию

Тестирование чёрного ящика на уровне API. Запросы отправляются через Postman (ручные коллекции) и/или curl. Применяемые техники тест-дизайна:

- **Классы эквивалентности** — разбиение входных данных на валидные/невалидные группы.
- **Граничные значения** — проверка краёв диапазонов (quantity = 0, 1, очень большое значение).
- **Таблица решений** — для бизнес-правил создания заказа.
- **Негативное тестирование** — заведомо некорректные данные и нарушение контракта.

Виды тестирования: смоук, функциональное, регрессионное.

## 5. Тестовое окружение

| Параметр | Значение |
|---|---|
| Запуск стека | `docker-compose up --build` |
| Базовый URL API | `http://localhost:8000` |
| Документация API | `http://localhost:8000/docs` |
| Инструменты | Postman, curl, браузер |
| Тестовые данные | Заполняются скриптом `seed.py` при старте |

## 6. Критерии входа и выхода

**Критерии входа:**
- Стек поднят, API отвечает на `GET /health` кодом 200.
- Тестовые данные загружены (`seed.py` отработал).

**Критерии выхода:**
- Все тест-кейсы выполнены (статус Passed/Failed проставлен).
- Нет открытых дефектов уровня Critical/Blocker.
- Дефекты Major описаны и переданы; принято решение по каждому.
- Смоук-чек-лист проходит на 100%.
